# Data Structure Guide for Coding Agents

> **Quick Start:** When users run commands like `/application-overview`, create files in `data/` following the structure below.

## Overview

The Az Infra Harness is a **read-only UI** that displays content generated by coding agents. When users run slash commands in a coding agent (OpenCode, Claude Code, or GitHub Copilot), the agent creates markdown and JSON files that the UI automatically displays.

## Supported Agents

| Agent | Config Location | Commands |
|-------|----------------|----------|
| OpenCode | `.opencode/commands/` | `/application-overview`, `/generate-code-bicep`, etc. |
| Claude Code | `.claude/commands/` | `/application-overview`, `/generate-code-bicep`, etc. |
| GitHub Copilot | `.github/prompts/` | `@workspace /application-overview`, etc. |

All agents share the same skill definitions in `skills/`. Each agent's config contains thin wrappers that reference these shared skills.

## How It Works

### User Flow
1. User opens the app at `/application-definition`
2. Sees empty boxes with commands: `/application-overview`, `/non-functional-requirements`, `/application-components`
3. Copies a command (e.g., `/application-overview`)
4. Pastes into their coding agent of choice
5. **Coding agent generates and saves file** — This is where you come in!
6. User refreshes browser
7. Content appears in the UI with proper formatting

### Coding Agent Responsibilities

When a user runs a slash command:
1. **Read** the corresponding skill file from `skills/`
2. **Follow** the skill instructions to gather information and generate content
3. **Save** to the correct file path in `data/`
4. **Follow** the naming conventions and formats in `DATA-STRUCTURE.md`

## Section 1: Application Definition

### Command → File Mapping

| User runs | Agent creates | Format |
|-----------|---------------|--------|
| `/application-overview` | `data/application-definition/application-overview.md` | Markdown |
| `/non-functional-requirements` | `data/application-definition/non-functional-requirements.md` | Markdown |
| `/application-components` | `data/application-definition/components/*.json` | JSON (one file per component) |

### File Formats & Templates

#### 1. application-overview.md

**Purpose:** High-level application overview

**Template:**
```markdown
# [Application Name]

## Description
[2-3 sentences describing what the application does and who uses it]

## Purpose
[Why this application exists - what business problem it solves]

## Key Features
- Feature 1 description
- Feature 2 description
- Feature 3 description
- Feature 4 description
- Feature 5 description
```

**Example:** See `data/application-definition/application-overview.md`

#### 2. non-functional-requirements.md

**Purpose:** Technical requirements (scale, security, performance, etc.)

**Template:**
```markdown
# Non-Functional Requirements

## Scale
- **Expected Users**: [total registered users, daily active users]
- **Concurrent Users**: [peak concurrent]
- **Growth**: [% year-over-year]
- **Data Volume**: [current size, projected growth]

## Availability
- **Target Uptime**: [percentage or SLA]
- **Maintenance Windows**: [when acceptable]
- **Critical Period**: [when 100% uptime is required]

## Security & Confidentiality
- **Data Sensitivity**: [PII, financial data, etc.]
- **Compliance**: [GDPR, HIPAA, SOC 2, etc.]
- **Authentication**: [MFA, SSO requirements]
- **Data at Rest**: [encryption requirements]
- **Data in Transit**: [TLS 1.2+, etc.]

## Integrity
- **Backup Frequency**: [daily, hourly, etc.]
- **Recovery Time Objective (RTO)**: [max acceptable downtime]
- **Recovery Point Objective (RPO)**: [max acceptable data loss]
- **Data Consistency**: [strong, eventual, etc.]

## Usage Patterns
- **Traffic Type**: [steady, bursty, predictable]
- **Peak Times**: [specific times/days]
- **Seasonal Variance**: [holiday patterns]
- **Geographic Distribution**: [regions where users are located]
```

**Example:** See `data/application-definition/non-functional-requirements.md`

#### 3. components/[name].json

**Purpose:** Define individual application components

**Template:**
```json
{
  "id": "component-id",
  "name": "Display Name",
  "type": "compute|data|networking",
  "description": "Brief description of what this component does"
}
```

**Valid `type` values:**
- `compute` - Web apps, APIs, containers, functions, background workers
- `data` - Databases, caches, blob storage, file shares
- `networking` - Load balancers, gateways, VNets, firewalls

**Naming conventions:**
- Filename should match `id` field
- Use lowercase with hyphens: `web-frontend.json`
- One component per file

**Examples:**

`components/web-frontend.json`:
```json
{
  "id": "web-frontend",
  "name": "Web Frontend",
  "type": "compute",
  "description": "React-based single-page application that provides the customer-facing interface for account management, order tracking, and support"
}
```

`components/api-backend.json`:
```json
{
  "id": "api-backend",
  "name": "API Backend",
  "type": "compute",
  "description": "RESTful API service that handles business logic, integrates with on-premises systems, and serves data to the web frontend"
}
```

`components/customer-database.json`:
```json
{
  "id": "customer-database",
  "name": "Customer Database",
  "type": "data",
  "description": "Stores customer profiles, preferences, authentication data, and application state"
}
```

**More examples:** See `data/application-definition/components/`

## Implementation Guide for Coding Agents

### Creating application-overview.md

```javascript
// Pseudo-code for coding agent
async function handleApplicationOverview(userInput) {
  // 1. Gather information through conversation with user
  const appInfo = await askUserQuestions([
    "What is your application name?",
    "Describe your application in 2-3 sentences",
    "What is the main purpose/business problem it solves?",
    "What are the 5 key features?"
  ])

  // 2. Generate markdown content
  const content = `# ${appInfo.name}

## Description
${appInfo.description}

## Purpose
${appInfo.purpose}

## Key Features
${appInfo.features.map(f => `- ${f}`).join('\n')}
`

  // 3. Save to correct location
  const filePath = 'data/application-definition/application-overview.md'
  await fs.writeFile(filePath, content)

  // 4. Confirm to user
  console.log(`Created ${filePath}`)
  console.log("Refresh your browser to see the changes!")
}
```

### Creating non-functional-requirements.md

```javascript
async function handleNonFunctionalRequirements(userInput) {
  // Gather NFRs through conversation
  const nfrs = await askUserQuestions([
    "How many users do you expect?",
    "What are your availability requirements?",
    "What compliance requirements apply?",
    "What are your backup/recovery needs?",
    // ... etc
  ])

  // Generate markdown with all sections
  const content = generateNFRMarkdown(nfrs)

  // Save
  await fs.writeFile(
    'data/application-definition/non-functional-requirements.md',
    content
  )
}
```

### Creating components/*.json

```javascript
async function handleApplicationComponents(userInput) {
  // 1. Identify all components
  const components = await identifyComponents(userInput)

  // 2. Create components directory if needed
  await fs.mkdir('data/application-definition/components', { recursive: true })

  // 3. Save each component as separate JSON file
  for (const component of components) {
    const filename = `${component.id}.json`
    const filePath = `data/application-definition/components/${filename}`
    await fs.writeFile(filePath, JSON.stringify(component, null, 2))
  }
}
```
