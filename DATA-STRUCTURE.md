# Data Directory Structure for Coding Agents

This directory contains all the data files generated by coding agents to populate the Azure Infra Prompt Kit UI.

## Directory Structure

```
src/data/
├── application-definition/     # Section 1: Application Definition
│   ├── application-overview.md
│   ├── non-functional-requirements.md
│   └── components/
│       ├── component-1.json
│       ├── component-2.json
│       └── ...
│
├── context/                    # Section 2: Context (Future)
│   ├── infrastructure-landscape.md
│   ├── platform-setup.md
│   └── development-context.md
│
├── application-architecture/  # Section 3: Application Architecture (Future)
│   ├── architecture-diagram.md
│   ├── components/
│   └── deployment-strategy.md
│
├── architecture-decisions/    # Section 4: Architecture Decisions (Future)
│   └── adrs/
│       ├── adr-001.md
│       ├── adr-002.md
│       └── ...
│
└── export/                    # Section 5: Export (Future)
    └── config.json
```

## How It Works

### For Users
1. Navigate to a section in the UI
2. See empty planning boxes with commands like `/application-overview`
3. Copy the command
4. Paste into coding agent (e.g., OpenCode)
5. Coding agent generates the file and saves it to the correct location
6. Refresh the page to see the content

### For Coding Agents
When a user runs a command like `/application-overview`, the coding agent should:
1. Generate the appropriate markdown or JSON content
2. Save it to the correct file path
3. Use the file naming conventions below

## Section 1: Application Definition

### Commands and File Paths

| Command | File Path | Format |
|---------|-----------|--------|
| `/application-overview` | `src/data/application-definition/application-overview.md` | Markdown |
| `/non-functional-requirements` | `src/data/application-definition/non-functional-requirements.md` | Markdown |
| `/application-components` | `src/data/application-definition/components/*.json` | JSON (one file per component) |

### File Formats

#### application-overview.md
```markdown
# [Application Name]

## Description
[2-3 sentence description of the application]

## Purpose
[Why this application exists, what problem it solves]

## Key Features
- Feature 1
- Feature 2
- Feature 3
```

#### non-functional-requirements.md
```markdown
# Non-Functional Requirements

## Scale
- **Expected Users**: [number]
- **Concurrent Users**: [number]
- **Growth**: [percentage] year-over-year
- **Data Volume**: [size]

## Availability
- **Target Uptime**: [percentage]
- **Maintenance Windows**: [description]
- **Critical Period**: [time periods]

## Security & Confidentiality
- **Data Sensitivity**: [types of data]
- **Compliance**: [regulations]
- **Authentication**: [requirements]
- **Data at Rest**: [encryption requirements]
- **Data in Transit**: [encryption requirements]

## Integrity
- **Backup Frequency**: [schedule]
- **Recovery Time Objective (RTO)**: [time]
- **Recovery Point Objective (RPO)**: [time]
- **Data Consistency**: [requirements]

## Usage Patterns
- **Traffic Type**: [description]
- **Peak Times**: [when]
- **Seasonal Variance**: [description]
- **Geographic Distribution**: [percentages by region]
```

#### components/[component-name].json
```json
{
  "id": "unique-component-id",
  "name": "Display Name",
  "type": "compute|data|networking",
  "description": "What this component does"
}
```

**Valid component types:**
- `compute` - Application servers, containers, functions
- `data` - Databases, storage accounts, caches
- `networking` - Load balancers, gateways, VNets

## Creating Components

### Single Component
Create one JSON file per component in `src/data/application-definition/components/`

Example: `src/data/application-definition/components/web-frontend.json`
```json
{
  "id": "web-frontend",
  "name": "Web Frontend",
  "type": "compute",
  "description": "React-based single-page application that provides the customer-facing interface"
}
```

### Multiple Components
The `/application-components` command should create multiple JSON files:

```bash
src/data/application-definition/components/
├── web-frontend.json      # Compute: Web UI
├── api-backend.json       # Compute: API
├── background-worker.json # Compute: Background jobs
├── database.json          # Data: Primary database
├── cache.json             # Data: Redis cache
└── app-gateway.json       # Networking: Entry point
```

## File Naming Conventions

### Markdown Files
- Use lowercase with hyphens: `application-overview.md`
- Match the command name (without `/`): `/application-overview` → `application-overview.md`

### JSON Files (Components)
- Use lowercase with hyphens: `web-frontend.json`
- Should match the component `id` field
- One component per file

## Checking Your Work

### Required Files for Complete Section

Application Definition section is complete when these files exist:
- ✅ `application-overview.md` (must have content)
- ✅ `non-functional-requirements.md` (must have content)
- ✅ At least one file in `components/` directory

### Validation
The UI will show:
- Grey border + empty circle = file missing or empty
- Blue border + checkmark = file exists with content
- Continue button disabled = not all files present
- Continue button enabled = all files present

## Example: Complete Setup

```bash
# Starting from src/data/application-definition/

# Create overview
cat > application-overview.md << 'EOF'
# Customer Portal

## Description
A self-service customer portal for account management and order tracking.

## Purpose
Reduce support call volume by enabling self-service.

## Key Features
- Account management
- Order tracking
- Support tickets
EOF

# Create non-functional requirements
cat > non-functional-requirements.md << 'EOF'
# Non-Functional Requirements

## Scale
- **Expected Users**: 50,000 registered users
- **Concurrent Users**: 500 peak
...
EOF

# Create components
mkdir -p components

cat > components/web-frontend.json << 'EOF'
{
  "id": "web-frontend",
  "name": "Web Frontend",
  "type": "compute",
  "description": "React-based SPA"
}
EOF

cat > components/api-backend.json << 'EOF'
{
  "id": "api-backend",
  "name": "API Backend",
  "type": "compute",
  "description": "RESTful API service"
}
EOF

# Check results
ls -la
ls -la components/
```

## Troubleshooting

### "Box still shows as empty"
- Check file exists in correct location
- Check file has content (not empty)
- Check file name matches exactly (case-sensitive)
- Refresh the browser page

### "Components not showing"
- Check files are in `components/` subdirectory
- Check JSON is valid (use `cat file.json | jq` to validate)
- Check each file has required fields: `id`, `name`, `type`, `description`
- Check `type` is one of: `compute`, `data`, `networking`

### "Continue button still disabled"
- All three files must exist with content
- At least one component JSON must exist
- Try hard refresh (Cmd+Shift+R or Ctrl+Shift+R)
