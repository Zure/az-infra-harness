# Data Directory Structure for Coding Agents

This directory contains all the data files generated by coding agents to populate the Az Infra Harness UI.

│       ├── component-2.json

├── context/                    # Section 2: Context (Future)

│       ├── adr-002.md
│       └── ...
│
└── export/                    # Section 5: Code Generation (placeholder)
    └── README.md

## How It Works

### For Users
1. Navigate to a section in the UI
2. See empty planning boxes with commands like `/application-overview`
3. Copy the command
4. Paste into coding agent (e.g., OpenCode)
5. Coding agent generates the file and saves it to the correct location
6. Refresh the page to see the content

### For Coding Agents
When a user runs a command like `/application-overview`, the coding agent should:
1. Generate the appropriate markdown or JSON content
2. Save it to the correct file path
3. Use the file naming conventions below

## Section 1: Application Definition

### Commands and File Paths

| Command | File Path | Format |
|---------|-----------|--------|
| `/application-overview` | `data/application-definition/application-overview.md` | Markdown |
| `/non-functional-requirements` | `data/application-definition/non-functional-requirements.md` | Markdown |
| `/application-components` | `data/application-definition/application-components.md` | Markdown |

### File Formats

#### application-overview.md
```markdown
# [Application Name]

## Description
[2-3 sentence description of the application]

## Purpose
[Why this application exists, what problem it solves]

## Key Features
- Feature 1
- Feature 2
- Feature 3
```

#### non-functional-requirements.md
```markdown
# Non-Functional Requirements

## Scale
- **Expected Users**: [number]
- **Concurrent Users**: [number]
- **Growth**: [percentage] year-over-year
- **Data Volume**: [size]

## Availability
- **Target Uptime**: [percentage]
- **Maintenance Windows**: [description]
- **Critical Period**: [time periods]

## Security & Confidentiality
- **Data Sensitivity**: [types of data]
- **Compliance**: [regulations]
- **Authentication**: [requirements]
- **Data at Rest**: [encryption requirements]
- **Data in Transit**: [encryption requirements]

## Integrity
- **Backup Frequency**: [schedule]
- **Recovery Time Objective (RTO)**: [time]
- **Recovery Point Objective (RPO)**: [time]
- **Data Consistency**: [requirements]

## Usage Patterns
- **Traffic Type**: [description]
- **Peak Times**: [when]
- **Seasonal Variance**: [description]
- **Geographic Distribution**: [percentages by region]
```

---

#### 3. application-components.md

**Purpose:** Define application components as a list

**Note:** This file creates a high-level list of components. Detailed configuration for each component is added later in the application-architecture section via `/configure-component`.

**Template:**
```markdown
# Application Components

## [Component Name]
**Type:** Compute|Data|Networking

[Brief description of what this component does]

## [Component Name 2]
**Type:** Compute|Data|Networking

[Brief description]
```

**Valid component types:**
- `Compute` - Application servers, containers, functions, background workers
- `Data` - Databases, storage accounts, caches, blob storage
- `Networking` - Load balancers, gateways, VNets, firewalls
- Custom types are also supported based on your application needs

**Example:**

```markdown
# Application Components

## Web Frontend
**Type:** Compute

React-based single-page application that provides the customer-facing interface for account management, order tracking, and support.

## API Backend
**Type:** Compute

RESTful API service that handles business logic, integrates with on-premises systems, and serves data to the web frontend.

## Customer Database
**Type:** Data

Stores customer profiles, preferences, authentication data, and application state.

## Application Gateway
**Type:** Networking

Entry point for incoming HTTPS traffic with SSL termination, WAF protection, and routing to backend services.
```

**More examples:** See `data/application-definition/application-components.md`

## File Naming Conventions

### Markdown Files
- Use lowercase with hyphens: `application-overview.md`
- Match the command name (without `/`): `/application-overview` → `application-overview.md`

## Checking Your Work

### Required Files for Complete Section

Application Definition section is complete when these files exist:
- ✅ `application-overview.md` (must have content)
- ✅ `non-functional-requirements.md` (must have content)
- ✅ `application-components.md` (must have content)

### Validation
The UI will show:
- Grey border + empty circle = file missing or empty
- Blue border + checkmark = file exists with content
- Continue button disabled = not all files present
- Continue button enabled = all files present

## Troubleshooting

### "Box still shows as empty"
- Check file exists in correct location
- Check file has content (not empty)
- Check file name matches exactly (case-sensitive)
- Refresh the browser page

### "Components not showing"
- Check `application-components.md` file exists
- Verify markdown structure with h2 headings for each component
- Ensure each component has **Type:** line
- Check component types are valid

### "Continue button still disabled"
- All three markdown files must exist with content
- Try hard refresh (Cmd+Shift+R or Ctrl+Shift+R)
