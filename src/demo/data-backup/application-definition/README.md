# Application Definition Data

> 📁 This directory contains markdown files generated by coding agents to populate the Application Definition section.

## Quick Reference

| Command | Creates File | Purpose |
|---------|--------------|---------|
| `/application-overview` | `application-overview.md` | Application name, description, purpose, and key features |
| `/non-functional-requirements` | `non-functional-requirements.md` | Scale, availability, security, integrity, and usage patterns |
| `/application-components` | `application-components.md` | List of application components (compute, data, networking) |

## File Formats

### application-overview.md

**Template:**
```markdown
# [Application Name]

## Description
[2-3 sentence description of what the application does]

## Purpose
[Why this application exists, what business problem it solves]

## Key Features
- Feature 1
- Feature 2
- Feature 3
- Feature 4
- Feature 5
```

**Current example:** `application-overview.md`

---

### non-functional-requirements.md

**Template:**
```markdown
# Non-Functional Requirements

## Scale
- **Expected Users**: [total registered users]
- **Daily Active Users**: [typical daily users]
- **Concurrent Users**: [peak concurrent users]
- **Growth**: [expected % year-over-year]
- **Data Volume**: [current size, projected size]

## Availability
- **Target Uptime**: [percentage or SLA]
- **Maintenance Windows**: [when acceptable]
- **Critical Period**: [when uptime is most important]

## Security & Confidentiality
- **Data Sensitivity**: [types of sensitive data handled]
- **Compliance**: [regulations: GDPR, HIPAA, etc.]
- **Authentication**: [requirements: MFA, SSO, etc.]
- **Data at Rest**: [encryption requirements]
- **Data in Transit**: [TLS version, etc.]

## Integrity
- **Backup Frequency**: [schedule]
- **Recovery Time Objective (RTO)**: [maximum downtime]
- **Recovery Point Objective (RPO)**: [maximum data loss]
- **Data Consistency**: [requirements: strong, eventual, etc.]

## Usage Patterns
- **Traffic Type**: [steady, bursty, predictable, etc.]
- **Peak Times**: [when traffic is highest]
- **Seasonal Variance**: [holiday patterns, etc.]
- **Geographic Distribution**: [where users are located]
```

**Current example:** `non-functional-requirements.md`

---

### application-components.md

**Template:**
```markdown
# Application Components

## [Component Name]
**Type:** Compute|Data|Networking

[Brief description of what this component does]

## [Component Name]
**Type:** Compute|Data|Networking

[Brief description of what this component does]
```

**Valid types:**
- `Compute` - Web servers, APIs, containers, serverless functions, background workers
- `Data` - Databases (SQL, NoSQL), caches, blob storage, file shares
- `Networking` - Load balancers, application gateways, VNets, private endpoints

**Current example:** `application-components.md`

## For Coding Agents

When a user runs `/application-overview`, `/non-functional-requirements`, or `/application-components`, create the appropriate markdown files in this directory.

### Creating application-overview.md

```bash
# Save to: src/infra/application-definition/application-overview.md
cat > application-overview.md << 'EOF'
# Customer Portal

## Description
A self-service customer portal that allows users to manage their accounts, view order history, track shipments, and access support resources.

## Purpose
Reduce support call volume by enabling customers to handle common tasks themselves while providing 24/7 access to account information.

## Key Features
- Account management
- Order tracking
- Support tickets
- Knowledge base
- Document downloads
EOF
```

### Creating non-functional-requirements.md

```bash
# Save to: src/infra/application-definition/non-functional-requirements.md
cat > non-functional-requirements.md << 'EOF'
# Non-Functional Requirements

## Scale
- **Expected Users**: 50,000 registered users, ~5,000 daily active
- **Concurrent Users**: Peak of 500 concurrent users
- **Growth**: 30% year-over-year
- **Data Volume**: ~2GB current, growing to 10GB over 2 years

## Availability
- **Target Uptime**: 99.5% (allows ~3.6 hours downtime per month)
...
EOF
```

### Creating application-components.md

```bash
# Save to: src/infra/application-definition/application-components.md
cat > application-components.md << 'EOF'
# Application Components

## Web Frontend
**Type:** Compute

React-based single-page application that provides the customer-facing interface for account management, order tracking, and support.

## API Backend
**Type:** Compute

RESTful API service that handles business logic, integrates with on-premises systems, and serves data to the web frontend.

## Customer Database
**Type:** Data

Stores customer profiles, preferences, authentication data, and application state.

## Application Gateway
**Type:** Networking

Entry point for incoming HTTPS traffic with SSL termination, WAF protection, and routing to backend services.
EOF
```

## Troubleshooting

### Box shows empty despite file existing
- Ensure file has content (not empty)
- Check file name exactly matches (case-sensitive)
- Verify file is in correct directory
- Hard refresh browser (Cmd+Shift+R or Ctrl+Shift+R)

### Continue button still disabled
- Need all 3 markdown files
- Check each file has actual content
- All boxes must show blue border before Continue is enabled
