## Purpose

This skill facilitates the creation of a structured markdown document that captures the network topology, landing zone configuration, existing resources, and key connectivity requirements for a solution. It aligns with the Azure Well-Architected Framework and Cloud Adoption Framework principles.

The generated file will reside at `src/data/context/infrastructure-context.md` and serve as the basis for the infrastructure's definition.

## Workflow

### Step 1: Ensure Prerequisites

Verify the existence of required files generated by prior skills or commands:

- **Application Definition:** `src/data/application-definition/application-overview.md`
  - Defines the application’s purpose, key features, and description.
- **Architecture Context (if generated):** `src/data/context/application-architecture.md`
  - Captures architecture-level diagrams, key components, and service boundaries.

If any file is missing or incomplete:

- ✅ Prompt the user to generate the missing files first:
  ```
  Prerequisite missing:
  - Ensure `application-overview.md` exists. Please run `/application-overview` to define the application’s definition.
  - Optionally, generate architecture context via `/application-architecture`.
  ```

### Step 2: Collaborative Infrastructure Information Gathering

Ask the user clear, individual questions (one topic at a time):

Important execution rules:

- Ask one logical group at a time and explicitly wait for answers before proceeding.
- Track each question and mark it as "answered", "partially answered", or "unanswered".
- If a question is skipped, vaguely answered, or ignored, re-ask it later for clarification.
- Do NOT proceed to file generation until all required sections contain actionable information.
- If the user provides incomplete input, summarize what is missing and explicitly request the remaining details.

#### **Network Topology**

- "What type of network topology do you envision (e.g., hub-and-spoke, virtual WAN, etc.)?"
- "Will there be connectivity to on-premises resources or other cloud environments? If yes, please specify."
- "What VNets and subnets do you anticipate, and what will they be used for?"

#### **Landing Zones**

- "Will you follow an enterprise-scale ‘CAF landing zone’ model? If yes, define the subscription and management group hierarchy."
- "How will you name resources (e.g., `rg-{workload}-{environment}-{region}-{instance}`)?"

#### **Connectivity Requirements**

- "What connectivity scenarios are required (e.g., private endpoints, ExpressRoute, VPN)?"
- "What are the primary inbound and outbound traffic routes (e.g., Azure Application Gateway, Azure Firewall)?"

#### **Existing Resources**

- "Are there existing resources to integrate (e.g., existing VNets, DNS zones, NSGs)?"
- "Will we need to provision additional infrastructure layers (e.g., subnets for specific workloads)? Please list them."

This skill MUST comply with the shared Interaction & Validation Standard:
`.opencode/skills/_shared/interaction-validation-standard.md`

### Step 3: Validate and Present Findings

Summarize the gathered data:

1. **Network Topology:**

- Type: [Hub-and-Spoke, vWAN, etc.]
- VNets, subnets: [List all identified ones, purpose]
- Connectivity plans: [Plans for on-prem/cloud integration]

2. **Landing Zone:**

- Model: [CAF model, enterprise-scale adherence?]
- Resource hierarchy/naming: [Defined naming conventions or grouping logic]

3. **Connectivity Requirements:**

- Governance strategy: Role of inbound and outbound gatekeeping systems.

Before moving to file creation:

- Validate that every question from Step 2 has a corresponding answer.
- If any required detail is missing (e.g., no topology type defined, no connectivity model specified), pause and ask a focused follow-up question.
- Present a short "Open Items" list if anything is still unclear.
- Only continue to Step 4 once all open items are resolved.

### Step 4: Save File & Validate Output

Generate the file following this structure:

```markdown
# Infrastructure Context

## Network Topology

- **Model**: [Topology Type]
- **VNets**: [VNets and purposes]
- **Connectivity**: [On-prem/cloud integration plans]

## Landing Zone

- **Model**: [CAF model adherence?]
- **Resource Groups**: [Hierarchy and naming conventions]

## Existing Resources

- **VNets**: [Existing VNets]
- **Subnets**: [Provisioned or planned subnets]
- **Networking**: [NSGs, Azure Firewall, etc.]

## Connectivity Requirements

- Inbound: [Inbound traffic routes]
- Outbound: [Outbound traffic destinations]
```

Validate the file exists at: `src/data/context/infrastructure-context.md` and is not empty.

### Step 5: Inform User + Next Steps

Let the user know:

```
✅ The infrastructure context has been successfully created!

📄 Location: src/data/context/infrastructure-context.md

🚀 Next, you can use `/platform-context` to define the platform-level requirements.
```

### Success Criteria

- File is created at the target location: `src/data/context/infrastructure-context.md`.
- File aligns with the structure outlined above.
- Validation ensures the file is not empty or malformed.
- User is informed of successful creation and provided with a follow-up command.
- All questions from Step 2 were explicitly answered or clarified before file generation.
- No required infrastructure section contains placeholders due to skipped user input.
